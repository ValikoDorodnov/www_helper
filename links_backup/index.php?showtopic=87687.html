GRASP (General Responsibility Assignment Software Patterns)

Это паттерны проектирования, используемые в ООП для решения общих задач по назначению обязанностей классам и объектам.

Или проще - шаблоны распределения обязанностей.

Всего их девять:

Основные
1. Information Expert (Информационный эксперт)
2. Creator (Создатель)
3. Controller (Контроллер)
4. Low Coupling (Низкая связанность)
5. High Cohesion (Высокая связность)

Дополнительные:

6. Pure Fabrication (Чистое производство или Чистая выдумка)
7. Indirection (Посредник)
8. Polymorphism (Полиморфизм)
9. Protected Variations (Сокрытие реализации)

Оглавление
Information Expert (Информационный эксперт)

Самый простой паттерн. Он гласит, что нужно назначать обязаннасти классу, который имеет самое большое количество информации для решения поставленной задачи.

Парадокс. Этот паттерн настолько очевиден, что его труднее всего понять. Потому что не можешь поверить, что все так просто и ищешь тайный смысл. А смысл его можно выразить одной фразой - "нельзя вырезать гланды через задницу".

Разберем на примере сферического в вакууме форума. Для наглядности сначала сэмулируем базу данных:
class DataBase
{   
    public $sections = array( 1 => 'ABC-Framework', 
                              2 => 'ABC-Flame'
                            );
    
    public $topics   = array( 1 => array(1 => 'SOLID', 
                                         2 => 'GRASP'
                                        ),
                            
                              2 =>  array(1 => 'Флуд')
                            );
    
    public $content  = array(
                              1 =>  array(1 => 'SOLID', 
                                          2 => 'GRASP', 
                                          3 => 'DRY'
                                          ),
                            
                              2 =>  array(1 => 'Information Expert', 
                                          2 => 'Creator', 
                                          3 => 'Controller'
                                          )
                            );                                        
}

Теперь напишем классы для получения информации:
class Section
{ 
    public function get($db)
    {
        return $db->sections;
    }
}

class Theme
{   
    public function get($db)
    {
        return $db->topics;
    }
}

class Message
{   
    public function get($db)
    {
        return $db->messages;
    }
}

А теперь сам форум:

class Forum
{
    public $db,
           $section,
           $theme,
           $message;

    public function __construct($db, $section, $theme, $message)
    {
        $this->db = $db;
        $this->section = $section;
        $this->theme   = $theme;
        $this->message = $message;
    }
    
    public function getSection($id)
    {
        $sections = $this->section->get($this->db);
        return $sections[$id];
    }

    public function getTheme($id_theme, $id)
    {
        $themes = $this->theme->get($this->db);
        return $themes[$id_theme][$id];
    }    
    
    public function getMessage($id_messages, $id)
    {
        $messages = $this->message->get($this->db);
        return $messages[$id_messages][$id];
    }  
}

///////////////////////////////////////////////////////////
$db = new DataBase;
$section = new Section;
$theme   = new Theme;
$message = new Message;
$forum   = new Forum($db, $section, $theme, $message);

echo $forum->getSection(1);
echo '<br>';
echo $forum->getTheme(1, 2);
echo '<br>';
echo $forum->getMessage(2, 1);

А теперь допустим нам нужно подсчитать общее количество сообщений на форуме. Какому классу назначить обязанность его подсчитать?

Информационным экспертом можно посчитать два класса. Сам форум (он знает все обо всем) и класс Message, у него конкретная информация.

Однако исходя из практицизма, ответственность нужно назначить последнему, так как это количество может понадобиться не только форуму, но и админке допустим. Так что делаем так:
class Message
{   
    public function get($db)
    {
        return $db->messages;
    }
// Назначаем ответственность    
    public function countAll($db)
    {
        $total = 0;
     
        foreach ($db->messages as $parent => $mess) {
            $total += count($mess); 
        }
    
        return $total;
    }    
}

И сможем подсчитать сообщения, не инициируя объект форума.
$db = new DataBase;
$message = new Message;
echo '<br>Всего сообщений :'. $message->countAll($db);


Это не помешает и форуму получить количество сообщений, в нем уже есть объект класса Message.

А вот если мы захотим получить количество тем, и станем делать это в классе Message, то получится черти-чё спагетти-код.

Вот выбор оптимального места, откуда получить информацию, и есть паттерн Information Expert.

Оглавление
Creator (Создатель)

Шаблон Creator решает, кто должен создавать объект. Фактически, это применение шаблона Information Expert к проблеме создания объектов. Information Expert решает, кто будет читать, Creator решает, кто будет создавать объект.

Более конкретно: нужно назначить классу B обязанность создавать экземпляры класса A, если выполняется как можно больше из следующих условий:

1. Класс B содержит или агрегирует объекты A.
2. Класс B записывает экземпляры объектов A.
3. Класс B активно использует объекты A
4. Класс B обладает данными инициализации для объектов A.

Посмотрим на примере. Вот решили мы собрать свой велосипед ;) Нам понадобятся как минимум три класса, которые делают раму, педали и колеса.

class Frame
{   
    public function get()
    { 
        return 'рама';
    }
}

class Pedal
{   
    public function get()
    {
        return 'педаль ';
    }
}

class Wheel
{   
    public function get()
    {
        return 'колесо ';
    }
}

Теперь изготовим сам велосипед:

class Bicycle
{  
    public function create()
    {
        $frame = new Frame;
        $pedal = new Pedal;
        $whell = new Wheel;
        
        return $frame->get() .' + '. $pedal->get() .' + '. $whell->get();
    }    
}

$bicicle = new Bicycle();
echo $bicicle->create();


Всё отлично: получилось. Однако какой то он не ахти:



Нужен другой, с двумя колесами. Потом третий понадобится, с тремя. Потом с одним большим, вторым маленьким. С квадратными колесами, без колес... Велосипедостроение - вещь безграничная. Если мы на каждый тип велосипеда создадим такой класс, то мы создадим сильную связанность. А это моветон. Дело в том, что нам может понадобиться сменить производителя колес (использовать другой класс). И нам везде придется все переписывать. Решается с помощью креатора "фабрика". Вот так:

class BicycleFactory
{  
    public $frame;
    public $pedal;
    public $whell;
    
    public function __construct()
    {
        $this->frame = new Frame;
        $this->pedal = new Pedal;
        $this->whell = new Wheel;
    }    
}

Тоесть он взял на себя функцию создания объектов. Поместил их в свойства. Теперь мы можем просто взять объект этого креатора и передать в конкретный класс велосипеда:

class Bicycle
{  
    public function create($creator)
    {
        $frame = $creator->frame;
        $pedal = $creator->pedal;
        $whell = $creator->whell;
        
        return $frame->get() .' + '. $pedal->get() .' + '. $whell->get();
    }    
}

$bicicle = new Bicycle();
// Передаем скреативленный набор объектов
echo $bicicle->create(new BicycleFactory);


Теперь мы можем создать целую кучу классов по созданию разных велосипедов, не опасаясь, что придется в 100 местах менять класс, реализующий колеса. Достаточно изменить в одном месте.

Этот класс отвечает первому требованию, он агрегирует (объединяет) в себе несколько объектов и хранит их в свойствах. Его называют "абстрактная фабрика".

Но паттерн проектирования "создатель" не ограничивается одним типом реализации. Поэтому креаторм может быть любой класс, отвечающий требованиям. Допустим рассмотрим последий пункт:
Класс B обладает данными инициализации для объектов A.

Классический пример - класс обертка над драйвером СУБД PDO.

class PDOconnect
{ 
    public $dbh;
 
    public function __construct($dsn, $user, $password)
    {
        try {
            $this->dbh = new PDO($dsn, $user, $password);
        } catch (PDOException $e) {
            echo 'Подключение не удалось: ' . $e->getMessage();
        }
    }
}


Это самый настоящий креатор, он создает объект для дальнейшего многократного использования. Это оптимальное решение, иначе пришлось бы в каждой модели создавать свой объект PDO.
И он отвечает требованиям, так как в нем содержатся данные для конфигурации объекта.

Креаторм можно назначить любой класс, по принципу информационного эксперта, если он в большей мере удовлетворяет каким то из этих требований:

1. Класс B содержит или агрегирует объекты A.
2. Класс B записывает экземпляры объектов A.
3. Класс B активно использует объекты A
4. Класс B обладает данными инициализации для объектов A.

Если класс создает объект другого класса, он априори креатор. Но нужно с ними поосторожнее, дабы не нарваться на ситуацию с велосипедом и не устроить сильную связанность системы.

Controller (Контроллер)

С этим паттерном проектирования так или иначе знаком любой программист, написавший чуть больше "Привет, Мир!". Если скрипт опирается на полученные от пользователя данные, это контроллер. Самый простой контроллер:

<?php

    switch (@$_GET['page']) {
  
        case 1 :
            echo 'Первая страница';
        break;
        
        case 2 :
            echo 'Вторая страница';
        break;
        
        default :
            echo 'Главная страница';
    }
	
/////////////////////////////////////////////////////// 
 
?>
 <br>
<a href="?">main</a> 
<a href="?page=1">1</a> 
<a href="?page=2">2</a>


Однако по неписанным правилам контроллер не должен сам выполнять никаких действий, а должен только управлять компонентами программы (делегировать им обязанности). Сам он бизнес-логики содержать не должен.

С легкой руки Падрика Брэди, описывающего паттерн MVC, контроллеры называют тонкими или толстыми, или даже ТТУК (Толстые тупые уродливые контроллеры). Если контролер содержит операции по обработке данных, ему пора на диету.

Вот пример ТТУК

        if (file_exists('./messages.txt')) {
            echo file_get_contents('./messages.txt');

}

    $controller = new Controller;
    $controller->actionIndex();
    
/////////////////////////////////////////////////////
?>
 <br>
<form action="" method="post">
<input name="message" type="text" value="" />
<input type="submit" value="Go" />

А вот так канонично:
class Controller

    public function actionIndex()

        $model = new Model;
        $view  = new View;

}


Контроллер сам ничего не делает с данными. Только принимает их и отправляет в нужную модель. А так же управляет модулем отображения.

Оглавление
Low Coupling (Низкая связанность)

Это принцип распределения обязанностей, который помогает делать части системы зависимыми от от наименьшего количества других частей. Тоесть организовать слабое зацепление модулей.

Для того, чтобы понять, что есть связанность, нужно уяснить, что такое зацепление.

Примером зацепления из жизни может служить какой-нибудь завод. Если директор завода дает указания начальнику цеха и с него жестко спрашивает выполнение плана, то это слабое зацепление. А если он пойдет раздавать указания каждому рабочему и контролировать работу каждого из них, это будет крайне неэфективное сильное зацепление. Особенно плохо, если еще и рабочие начнут бегать из цеха в цех за деталями и инструментами.

Низкая связанность, это организация рабочего процесса так, чтобы директору оставалось как можно меньше работы. :) И чтобы каждый выполнял свою работу, не бегая в другой цех.

Допустим мы хотим сделать текстовый редактор. Для него понадобилась библиотека обработки строк. В ней основной класс и два вспомогательных: один отвечает за установку кодировки, второй за преобразование регистра. Вот в таком виде приложение имеет все признаки нарушения принципа слабой связанности:

class WordProcessor
{ 
    protected $case = 'lower';
	
    public function setCharacterCase($case)
    {
        $this->case = $case;
    }

    public function getConvertedText($string)
    {// Один класс библиотеки
	    $encoding = new Encoding;
	    $encoding->setUTF8();
	 // Второй класс библиотеки
        $capsloc = new CapsLock;
		return ($this->case === 'lower') ? $capsloc->getLower($string) : $capsloc->getUpper($string);
    }
}
//  САМА БИБЛИОТЕКА    
///////////////////////////////////////////////////////////////    
class StringConversion 
{ 
    public function setUTF8()
    {
	    $encoding = new Encoding;
	    $encoding->setUTF8();
    }
	
    public function caseString($string, $case = 'lower')
    {
        $capsloc = new CapsLock;
        return ($case === 'lower') ? $capsloc->getLower($string) : $capsloc->getUpper($string);
    }     
}    
    
class Encoding
{    
    public function setUTF8()
    {
        mb_internal_encoding("UTF-8");
    }
}

class CapsLock 
{ 
    public function getLower($string)
    {
        return mb_strtolower($string);
    }
	
    public function getUpper($string)
    {
        return mb_strtoupper($string);
    }    
}    
    
///////////////////////////////////////////////////////////////////
	$string = 'привет, мир!';
	
	$processor = new WordProcessor;
	$processor->setCharacterCase('upper');
	echo $processor->getConvertedText($string); // ПРИВЕТ, МИР!


А если мы будем строить работу через управляющий класс (единую точку входа), то принцип будет соблюден:


class WordProcessor
{ 
    protected $case = 'lower';
	
    public function setCharacterCase($case)
    {
        $this->case = $case;
    }

    public function getConvertedText($string)
    {// Теперь редактор понятия не имеет, какой класс либы за что отвечает
     // Все обязанности дилегированы управляющему классу
	    $convert = new StringConversion;
	    $convert->setUTF8();
        return $convert->caseString($string, $this->case);
    }
}


Тепрь намного проще сменить библиотеку (можно легко написать адаптер), и в коде разобраться гораздо проще, так как название управляющего класса как правило совпадает с названием всей библиотеки. А искать разрозненные классы, особенно если редактор использует несколько библиотек - та еще история.

Оглавление
High Cohesion (Высокая связность)

Этот принцип задаёт свойство сильной связности внутри подсистемы.

Может показаться, что это входит в когнитивный диссонанс с Low Coupling (принципом низкой связанности), но ключевое слово тут "подсистема". Подсистемой нужно считать класс.

Возьмем тот же завод, что рассматривался в предыдущем объяснении. Нарушение принципа высокой связанности, это когда слесарь Вася, вместо того, чтобы попросить сварщика Петю, сам возьмет держак и "присрет" сикось-накось какую-нибудь деталь.

Допустим мы имеем класс калькулятора: 

class Сalculator
{ 
    public function сalculate($value1, $value2, $operator)
    {
        $list = array('+', '-', '*', '/');
     
        if (!in_array($operator, $list)) {
            throw new Exception("it's not the operator");
        }
      
        if ($value2 == 0 && $operator === '/') {
            throw new Exception("Division by zero"); 
        }
        
        eval('$total = '. $value1 . $operator . $value2 .';');

1. Трудность понимания.
Этот код трудно читать без комментариев. Конкретно этот еще можно осилить, тут все элементарно. Но если будет портянка побольше, алгоритмы посложнее, то будет болото.

2. Сложность при повторном использовании.
Если мы захотим этот класс использовать для других схожих целей, выйдет казус. Допустим при вычислениях в программе выброс исклчений - самое то. А если мы захотим использовать его для пользовательского калькулятора, то нужен другой механизм обработки ошибок.

3. Сложность поддержки.
Если потребуется произвести более сложные вычисления с несколькими значениями и операторами, придется полностью переписывать весь метод.

4. Ненадежность, постоянная подверженность изменениям.
Список операторов статичен и не подвержен изменениям извне. А значит мы лишаемся возможности использовать другие операторы (то же остаток от деления '%' к примеру).

class Сalculator
{
    public $valid_operators = array('+', '-', '*', '/');    
    public $all_operators   = array('+', '-', '*', '/', '%');
	
    protected $user_error   = false;
    protected $text_error   = '';
    
    public function setUserErrors()
    {
        $this->user_error = true;
    }    

    public function сalculate($value1, $value2, $operator)
    {   
	    // Пример возможного дальнейшего развития
	    /*
        if ($this->validData($value1)) {
            return $this->errorHandler("The data is not valid"); 
        }
        */
		
        $result = $this->validOperators($operator);		
		
        if($result === true) {
            eval('$total = '. $value1 . $operator . $value2 .';'); 
            return $total;
        } 
        else {
            return false;
        }
    }     
    
    protected function validOperators($operator)
    {  
        if (!in_array($operator, $this->valid_operators)) {
            return $this->errorHandler("It operator is banned");
        }
        
        if (!in_array($operator, $this->all_operators)) {
            return $this->errorHandler("It's not the operator");
        }
        
        return true;
    } 

    public function errorHandler($message)
    {
        if($this->user_error) {
            $this->text_error = $message;
            return false;            
        }
        else {
            throw new Exception($message);
        }
    } 

    public function getTextError()
    {  
        return $this->text_error;
    }     
}

Оглавление
Pure Fabrication (Чистая выдумка)

Чистая выдумка, это класс, не отражающий никакого реального объекта предметной области.

Понять паттерн на примере из жизни очень сложно, выдумка, она и есть выдумка. Ничего материального. Так что перейдем сразу к примеру из практики программирования.

Допустим мы делаем интернет-магазин. Наделали разделов, товарв, учетов и так далее. Затестили - все хорошо. Пора зарабаывать денежки. Для этого нужно сделать модуль для работы с платежными системами.

А как с этим модулем работать? Как внедрить его в систему? Ведь нужно как то организовать хранение полученных с биллинга данных. Прописывать каждому API методы для работы с СУБД - получится явное нарушение принципов Low Coupling и High Cohesion.

Первого - потому что класс работы с биллингом не должен работать и с базой данных. Получается сильное зацепление. Зацепится он за класс базы данных.

Второго - потому что получится довольно большая обязанность у метода. Он как-то должен и коннект организовать, и данные подготовить, и запись осуществить, и много еще чего может понадобиться. А если разносить на разные методы, то еще хуже. Задача для класса непрофильная и их может оказаться больше, чем основных. Тогда класс API потеряет свою узкую специализацию, начнет работать на два фронта, и мы нарушим Принцип единственной обязанности SOLID.

Но выход есть: как раз вот этот паттерн, чистая выдумка.

Мы сделаем общий класс, который возьмет на себя обязанность сохранять данные из классов биллингов в СУБД.

А раз это класс, там можно сделать много методов. Принцип Low Coupling будет соблюден, этот класс освободит классы биллингов от непрофильных действий. А так же избежим нарушения High Cohesion, разнеся сопутствующие сохранению обязанности по разным методам общего класса. Ну если таковые понадобятся конечно.

Indirection(Посредник)

Этот шаблон решает проблему прямой связности. Решается введением промежуточного объекта для обеспечения связи между частями программы, которые не связаны между собой напрямую.

Посредник, он и в Африке посредник. Тут и объяснять особо нечего. Пример из Pure Fabrication, это частный случаей и этого паттерна, так как он посредник между классом биллинга и классом базы данных.

Так же вариантами посредника являются шаблоны Adapter, Facade, Observer.

Вот рассмотрим пример адаптера на тех же биллингах. Адаптер решает проблему адаптации разнородных классов в одну систему. Допустим мы не сами написали классы терминалов, а набрали у платежных систем готовых API. В них уже есть методы для получения данных. Но они все разные, называются по разному, принимают разные аргументы и так далее. А менять код API крайне не желательно, нарушится преемственность версий. Вот тут и нужен посредник между системой и этой кучей разнородных классов.

Ну примерно так:

Polymorphism (Полиморфизм)

Полиморфизм позволяет обрабатывать альтернативные варианты поведения на основе типа и заменять подключаемые компоненты системы.

Рассмотренный выше пример адаптера является частным случаем полиморфизма. Потому что там к общей системе адаптировалось сразу несколько разных биллингов. А раз они разные, значит полиморфные.

Другой, более продвинутый пример полиморфизма: с абстрактным классом. Абстрактный класс, это класс, который содержит хотя бы один абстрактный метод. Абстрактный метод, это пустой метод, без реализации.

И пример. Допустим у нас завод по производству автомобилей. И есть линейка моделей с разными кузовами, но одинаковыми двигателями. И есть общий склад, с которого продаются эти автомобили.

Так вот чтобы на каждой линии сборки не делать полный цикл производства моторов, делается один общий цех, откуда эти моторы распределяются по линиям сборки.

Сам завод не знает, на какую модель конкретно пойдет двигатель, но точно знает, что на какую-то пойдет. Вот это определение"какая-то" и есть абстракция.

Такая абстракция позволяет не наследнику обращаться к базовому классу, а наоборот. Базовый класс может использовать метод какого-нибудь из наследников. А раз наследников может быть много, то это и есть полиморфизм в чистом виде. 

abstract class VAZ

Protected Variations (Сокрытие реализации)

Основная проблема, решаемая шаблоном Protected Variations: как спроектировать объекты, подсистемы и систему, чтобы изменение этих элементов не оказывало нежелательного влияния на другие элементы.

Что-то из области "не трогай какушку пока сухая, она и не завоняет".

Это очень широкий паттерн, его сложно показать на примере. Нужно стараться проектировать систему так, чтобы если в одном месте что-то изменили, то эти изменения не вылезли боком на другом конце.

Существует много принципов проектирования, которые тем или иным образом являются проявлением этого шаблона:

- Проектирование на основе данных
- Поиск служб
- Проектирование на основе интерпретаторов
- Рефлексивное проектирование или проектирование на метауровне
- Унифицированный доступ
- И т.д.

Есть для этого и куча инструментов, препятствующих казусам. Инкапсуляция, интерфейсы, полиморфизм, перенаправление и т.д. и т.п. и пр. пр.

Главное обеспечить устойчивость интерфейсов, это основная идея паттерна.
